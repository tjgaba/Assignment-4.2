## Overview
This reflection explains how documentation and collaboration were handled while working on the Conference Room Booking System. The main goal of this task was not just to describe system features, but to make sure the project is well-documented, easy to understand, and simple for others to contribute to.

---

## Documentation Approach
We focused on keeping documentation clear, organised, and easy to navigate.

Some of the key things we did include:
- Structuring the README so it clearly explains what the project is, why it exists, and how to contribute
- Organising files into meaningful folders such as `artifacts/` and `ceremonies/` instead of placing everything in one location
- Writing user stories using a consistent format that separates:
  - Who the user is
  - What they want to do
  - Acceptance criteria
- Improving layout and spacing so documents are easier to read and scan

This made the documentation easier to follow for both technical and non-technical readers.

---

## Collaboration Practices
Collaboration was handled using simple but effective rules to avoid confusion and conflicts.

Key practices included:
- Using clear branch naming conventions (`feature/`, `bugfix/`, `docs/`, `issues/`) so everyone knows what a change is about
- Making Pull Requests the only way to merge changes into the main branch
- Using a Pull Request template to guide contributors on what information to include
- Linking back to sprint documents, user stories, or issues for better traceability

These steps helped keep collaboration organised and made reviews easier and more consistent.

---

## Issue Tracking and Improvement
Instead of hiding problems, known issues and edge cases were clearly documented using GitHub Issues on the Read Me file.

This helped by:
- Making known problems visible to contributors on the Read Me file
- Giving contributors clear tasks to work on
- Encouraging fixes through labelled issues such as `good first issue`
- Keeping one clear place where problems are tracked and resolved

This approach supports gradual improvement and keeps the project realistic and honest.

---

## Reflection on the Outcome
This task showed that good documentation and collaboration matter just as much as writing code. By taking time to structure documents properly and define clear workflows, the project became easier to understand, review, and extend.

The work done here sets a solid foundation for future development and makes it easier for new contributors to join the project.

---

## Conclusion
Overall, this task helped reinforce real-world software development practices. Clear documentation, structured collaboration, and transparent issue tracking all contribute to a healthier and more maintainable project as it continues to grow.
